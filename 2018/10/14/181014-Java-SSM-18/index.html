  • 事务原本是数据库中的概念，在dao层，但一般情况下，需要将事务提升到业务层，即service层。

Spring事务管理API

Spring的事务管理主要用到两个事物相关的接口: PlatformTransactionManager事务管理接口，TransacitonDefinition事务定义接口

事务管理接口

  • 事务管理器是PlatformTransactionManager接口对象。其主要完成事务的提交，回滚，以及获取事务的状态信息。
常用的两个实现类
  • DataSourceTransactionManager: 使用JDBC或iBatis进行持久化数据时使用
  • HibernateTransactionManager: 使用Hibernate进行持久化数据处理
Spring回滚方式
  • Spring事务的默认回滚方式是： 发生运行时异常回滚，发生受查异常时提交
  • 不过，对于受查异常，程序员也可以手工设置其回滚方式

事务定义接口

  • TransactionDefinition中定义了事物描述相关的三类常量：事务隔离级别，事务传播行为，事务默认超时时限，以及对他们的操作
五个事物隔离级别常量

这些事物均是以ISOLATION开头，如： ISOLATION_XXX

  • DEFAULT: 采用DB默认的事物隔离级别。MySql的默认行为是REPEATABLE_READ; Oracle默认的是READ_COMMITTED
  • READ_UNCOMMITTED: 读未提交。未解决任何并发问题
  • READ_COMMITTED: 读已提交。解决脏读，存在不可重复读与幻读
  • REPEATABLE_READ: 可重复读。解决脏读，不可重复读，存在幻读
  • SERIALIZABLE: 串行化。不存在并发问题
七个事务传播行为常量

  • 事务传播行为是指，处于不同事务的方法相互调用时，执行期间事务的维护情况。如A事务中的方法doSome()调用B事务中的方法doOther(),在调用执行期间事务的维护情况。

  • 事务是传播行为是加在方法上的

  • 事务传播行为常量都是以PROPAGATION_开头，如PROPAGATION_XXX

  • REQUIRED
    指定的方法必须在事务内执行。若当前存在事务，就加入到当前事务中，若当前没有事务，则创建一新事务。（这种传播行为最常见，也是Spring默认的事务传播行为）

    如事务传播行为加在doOther()方法上，若doSome()方法在调用doOther()方法时就是在事务内运行的，则doOther()方法的执行也加入到该事务内执行。若doSome()方法在调用doOther()方法时没有在事务内执行，则doOther()方法会创建一个事务，并在其中执行。

  • SUPPORTS： 指定当前的方法支持事务，若当前没有事务，也可以以非事务的方式执行

  • MANDATORY: 指定的方法必须在当前事务执行，若当前没有事务，直接抛出异常

  • REQUIRES_NEW: 总是新建一个事务，若当前存在事务，就将当前事务挂起，直到新事务执行完毕。

  • NOT_SUPPORTED： 指定的方法不能在事务环境中执行，若当前存在事务，就将当前事务挂起。

  • NEVER 指定的方法不能在事务环境下执行，若当前存在事务，就直接抛出异常
  • NESTED 指定的方法必须在事务内执行，若当前存在事务，则在嵌套事务内执行。若当前没有事务，则创建一个新事物
定义了默认事务超时时限
  • TIMEOUT_DEFAULT定义了事物底层默认的超时时限，该值一般使用默认值即可

使用Spring的事务代理工厂管理事务

  • Github-项目地址
  • 为目标类，即Service类的实现类创建事务代理。事务代理使用的类是TransactionProxyFactoryBean.

  • 该类需要初始化以下属性：

    • transactionManager: 事务管理器
    • target： 目标对象，即Service实现类
    • transactionAttributes: 事务属性设置

  • 对于XML配置代理方式实现事务管理时，受查异常的回滚方式，程序员可以通过“-异常” 方式，可以使得发生指定的异常时进行回滚，通过“+异常”方式，使得发生指定异常时进行事务提交.
